>
> **Setup on a new PC:** see [`SETUP.md`](SETUP.md) — three layered install paths (raw `setup.ps1` / `bootstrap.ps1` fetch+chain / `Sentinel-Setup.exe` clickable), inline Telegram credential walkthrough, troubleshooting, uninstall.

## What This Is

Sentinel OS is a modular agentic engine that runs locally on constrained hardware (RTX 3050 4GB VRAM / 16GB RAM). It is NOT a job search tool — it is a general-purpose framework where skills are pluggable Python files, agents are YAML configs, and the system handles queuing, VRAM gating, logging, and execution. The job search domain is the first test case.

## Architecture Overview

```
User Input → Router (deterministic, no LLM) → Job Queue (SQLite) → Worker (async)
  → Orchestrator (dict lookup) → Agent (YAML-defined) → Skill Pipeline → Result → DB
```

## Key Design Decisions

- **Deterministic routing over LLM routing.** The router and orchestrator use zero LLM calls. String parsing and dict lookups only. VRAM is too scarce to waste on intent classification.
- **Fixed skill pipelines over ReAct.** Agents execute a predefined list of skills in order. No dynamic tool selection — 3B models can't do it reliably. The human defines the pipeline in YAML.
- **SQLite for everything.** Task queue, locks, and future knowledge base all use SQLite. Zero infrastructure, survives crashes, uses negligible RAM.
- **Ollama manages VRAM.** We set `keep_alive` and let Ollama handle model loading/unloading. No manual VRAM management.
- **Async worker with GPU lock.** Only one GPU task at a time. The lock is in the DB (not memory) so it survives crashes. Stale locks auto-recover on startup.
- **All paths from config.** Every file path, model name, threshold, and URL is in `core/config.py`. Never hardcode.
- **Teaching loop: Claude teaches Qwen, not replaces it.** When Qwen fails a coding task, the local `claude` CLI provides the fix + a teaching note + a reusable pattern. Qwen re-attempts. Success = pattern stored in knowledge base. Failure = Claude's answer returned, limitation recorded. Over time, Qwen solves more problems from the knowledge base alone, reducing CLI invocations.
- **Claude teaching is via the local `claude` CLI subprocess, NOT the Anthropic API.** Sentinel itself makes ZERO outbound network calls. The CLI handles auth via the user's existing Claude Code login. We invoke `claude.exe` directly (bypassing the .cmd shim to avoid Windows arg-quoting issues).
- **Knowledge base is SQLite FTS5, not a growing text file.** Capped at 500 entries, pruned by usage_count. Context injection into Qwen's prompt is capped at 4000 chars. Prevents context window bloat.
- **Executor validates all solutions.** Code is actually run, not just syntax-checked. Timeout = 30s.
- **Complexity routing is heuristic, not LLM-powered.** Keyword matching and input length, not an inference call. Zero VRAM cost for routing decisions.
- **Knowledge base informs routing.** If a problem class has a "limitation" entry (Qwen failed it before), the classifier bumps it to "advanced" tier immediately — don't waste time on models that have already proven they can't handle it.
- **Fallback chain: recommended → next-tier-up → claude-cli.** Always try the fastest local option first. Claude CLI is the ceiling, not the default.
- **Per-agent model pinning.** Agents can force a specific model via YAML config (`model: qwen-coder`), bypassing complexity routing. Useful for tasks where you know exactly which model works.
- **No basic-tier local model.** On constrained VRAM the model-swap penalty (5–10s to unload one and load another) wipes any speed gain from a smaller model. The simplified roster has two standard-tier locals (brain + coder) and one advanced (claude-cli) — no 1.5b.
- **Brain is Qwen 3 1.7B (`sentinel-brain`), not Qwen 2.5 3B.** Smaller model = faster loads + more VRAM headroom for the worker. Custom Modelfile bakes `num_ctx=8192`, `num_predict=1024`, `temperature=0.1`.
- **Brain uses `/no_think` mode.** Qwen 3's thinking adds 10+ s of latency. For intent classification we want fast decisive routing, not deep reasoning.
- **After GPU tasks, send `keep_alive: 0`.** Worker actively frees VRAM after each GPU-tagged job so the brain can reload immediately rather than waiting for the keep-alive timeout.
- **Authorization is non-negotiable.** Every Telegram handler checks `update.effective_user.id ∈ TELEGRAM_AUTHORIZED_USERS`. Bot is public; anyone could find it.
- **Three Telegram lanes**:  `/code` → Qwen Coder via skill pipeline + teaching loop.  `/claude` → Claude CLI direct subprocess passthrough.  Free text → Brain (Qwen 3 1.7B) → dispatches to agent → reports results.
- **Persona files are always loaded into the brain (Phase 10).** `IDENTITY.md`, `SOUL.md`, `USER.md`, `MEMORY.md` live in `workspace/persona/` — human-editable, git-tracked, hash-watched. Edits outside `file_guard.authorize_update()` ping Telegram. The brain reads all four every chat.
- **Three-tier memory (Phase 10).** Working = in-process per-session ring (20 msgs); episodic = per-agent FTS5 with decay/prune; semantic = global facts upsert-by-key with confidence-aware merge. New `memory.db` — does NOT touch `knowledge.db` (the code-pattern KB is for /code learnings, not user facts).
- **Auto-extraction (Phase 10).** Every 5 free-text messages in a Telegram session, the brain extracts durable facts in the background and stores them at confidence 0.6 (auto_extracted). They're visible to agents flagged `UNCONFIRMED`; nightly curation promotes them to 1.0 after owner approval.
- **Scheduler defers to the worker GPU lock (Phase 11).** Scheduler does NOT enforce its own concurrency cap on GPU work — it dispatches via `route()` and lets the worker queue + GPU lock serialize. `SCHEDULER_MAX_CONCURRENT=1` is a sanity belt, not the real serializer.
- **Internal maintenance bypasses the router (Phase 11).** `/internal_<name>` job commands resolve via `scheduler.INTERNAL_HANDLERS` and call async Python directly — saves the queue round-trip on pure SQLite/disk work (WAL checkpoint, backups, resource probe, file_guard heartbeat, auto-curation).
- **EST is the human-facing timezone (Phase 11).** Cron expressions and `--hours HH-HH` active-windows are interpreted in `America/New_York`. `next_run_at` is always stored UTC. `/schedule list` renders local.
- **Health endpoint stays on 127.0.0.1 (Phase 11).** `HEALTH_PORT=18700` bound to `HEALTH_BIND_HOST=127.0.0.1` only. Never exposed externally. `/dashboard` renders the same JSON for chat consumption from your phone.
- **Log rotation is Windows-tolerant (Phase 11).** `_WindowsTolerantRotatingHandler` swallows `PermissionError` on rollover (Windows can't rename a file held open by another process). Logs continue appending; rotation retries on next overflow. No-op on Linux/macOS.
- **Job-pipeline scoring is 1-5 weighted, not 0-1 binary (Phase 12).** Five dimensions (cv_match, north_star, comp, cultural_signals, red_flags) with PROFILE-derived weights. Bands: ≥4.5 apply_now, 4.0-4.4 worth_applying, 3.5-3.9 maybe, <3.5 skip. Hard cutover -- no backward compat with the old `recommend: bool`. Posting legitimacy is a SEPARATE signal (3 tiers, never affects the score).
- **Title + avoid filter runs BEFORE the LLM (Phase 12).** Each scraped posting is checked against PROFILE.title_filter (positive must match, negative rejects) plus PROFILE.avoid_companies + per-call `--avoid` keywords. Drops save Qwen GPU cost on guaranteed-rejection postings. Drop counts logged separately as `dropped_title` vs `dropped_avoid`.
- **Workplace filter is server-side where possible (Phase 12).** `--workplace remote|on-site` translates to jobspy's `is_remote` boolean for server-side filtering. `hybrid` and `all` get None (no server filter); the post-filter happens via `JobExtraction.workplace_pref` carry-through and the scorer penalizes mismatches.
- **Applications table is the lifecycle source of truth (Phase 12).** `sentinel.db::applications` keyed by `url_hash` (sha256 of canonicalized URL with utm_*/gclid/fbclid stripped). `upsert_application` is idempotent and never regresses an advanced state ('applied' stays 'applied' on re-scrape). Canonical states mirror career-ops: evaluated → applied → responded → interview → offer → rejected → discarded. Spanish aliases accepted on transition.
- **Hard commute gate before LLM scoring (Phase 12.5).** When PROFILE says `willing_to_relocate=False`, has both `primary_zip` and `onsite_max_miles`, and a posting is on-site/hybrid AND (in a foreign country OR > onsite_max_miles away), `skills/job_score.py` short-circuits to `score=2.0, recommendation='skip', red_flags=1` BEFORE the LLM call. Saves ~10s GPU per disqualified posting AND prevents the LLM's optimism bias from masking deal-breakers. Conservative on uncertainty: if the location can't be geocoded (state-only, "Remote", garbage), the gate does NOT fire — the LLM still gets to score it.
- **Scrape radius is profile-driven (Phase 12.5).** `skills/job_scrape.py` overrides jobspy's default `distance=250` with `profile.location.onsite_max_miles` when the candidate cannot relocate AND the workplace is on-site/hybrid. Indeed/etc. then return only nearby postings — saves bandwidth on top of LLM cost. Explicit `--distance N` from the CLI is honored.
- **LinkedIn fetches full description (Phase 12.5).** `linkedin_fetch_description=True` passed to `jobspy.scrape_jobs()`. Adds ~1s/posting but gives the scorer full body (~2000 chars) instead of the 150-char search snippet that previously made LinkedIn postings default to neutral-3 across all dimensions.
- **Scrape-time dedup against applications table (Phase 13).** Every scraped row's URL gets canonicalized (utm/gclid stripped, lowercased) and hashed; if `applications.url_hash` already has it, drop BEFORE the extract+score chain. Saves the entire LLM cost on dupes. Re-runs of the same query become near-free.
- **Per-archetype scoring weights (Phase 13).** `core/archetypes.ARCHETYPE_DEFAULT_WEIGHTS` defines 5-dim weight presets per archetype: RSM/TSM heavy on north_star (this IS the role), AE heavy on comp (don't take a step down), RevOps heavy on cv_match (skills-driven), CSM heavy on cultural_signals (relationship role), SDR heavy on red_flags (mill-job density). PROFILE override path: `target_roles.archetypes[*].weights` with auto-normalize-to-sum=1.0.
- **Region keyword nudges (Phase 13).** `PROFILE.title_filter.region_boost` / `region_avoid` are matched against title+location. Each hit is +/-0.5 capped at +/-1.0, applied to `north_star` AFTER the LLM call. Accepted-states whitelist similarly: state outside the list takes -0.5 on `cultural_signals`. Both are deterministic, no LLM cost. Unparseable locations fall through (no penalty applied to noise).
- **LLM batching with three-tier failure recovery (Phase 13).** `job_score` is `accepts_list=True` — pipeline hands it the full list of extracted postings; the skill chunks into `JOB_SCORE_BATCH_SIZE` (default 5) and does ONE LLM call per chunk. Per-item context (archetype, commute gate, region nudge, state whitelist, weights) is resolved BEFORE the batch, so commute-gated postings short-circuit and never count toward batch size. Failure recovery: (a) `LLMError` → per-item fallback for whole chunk, (b) malformed batch JSON / wrong-length results → per-item fallback, (c) one bad item in the array → per-item retry for THAT item only. Cuts a 40-posting `/jobsearch` from ~7 min to ~3.5 min.
- **Adaptive title filter is auto-apply, with safety bounds (Phase 13).** After every `/jobsearch`, the bot reads `logs/last_scrape_stats.json`. If `dropped_title >= JOB_ADAPTIVE_FILTER_MIN_DROPS` (10) AND `drop_ratio >= JOB_ADAPTIVE_FILTER_DROP_RATIO` (0.4), it samples 20 dropped titles, asks the brain for 1-3 candidate negatives, validates them against existing positive/seniority_boost (won't add anything that'd block target archetypes), atomically appends to `PROFILE.title_filter.negative`, and notifies the user via Telegram. Cap of 3 new negatives per run prevents runaway. Best-effort throughout — any failure is logged but never bubbles up to the `/jobsearch` user.
- **Query expansion is deterministic, not brain-based (Phase 13).** `core/query_expansion.ABBREV_MAP` hardcodes sales-title abbreviations (RSM ↔ Regional Sales Manager, AE ↔ Account Executive, etc.). Each `/jobsearch` runs jobspy 1-3× with variants; rows deduped by URL across variants before the title filter. Skipping the brain here avoids ~5s GPU model swap (qwen3-brain ↔ qwen2.5:3b) for jargon coverage that doesn't actually need an LLM. New jargon → add a row.
- **JobScrapeOutput must stay single-list-field (Phase 13).** The agent's `output_is_list=True` unwrap path requires exactly one list-typed field so the next skill's fan-out works. Phase 13 telemetry (dropped titles sample, query variants used) goes to a sidecar file `logs/last_scrape_stats.json` instead of being added to the typed output. ## Working with this user (handoff notes for the next assistant)

The user runs this project solo, on the constrained hardware described above, with strict standards. Match this collaboration style on day one:

- **Engineering bar.** Senior-engineer-quality work. Strict ECC (Edge Case Coverage) testing — every public function, state machine, and threshold gets an explicit test. Regression tests from ALL prior phases must stay green ("236/236 regression green" is the routine bar, not aspirational). Phases ship in BATCHES (e.g. Phase 16 = Batch 0, A, B, C, D, C-safety) so each is independently reviewable + revertible.
- **Communication style.** Terse over verbose. No fluff, no preamble, no end-of-response recap of what just happened — they read the diff. Direct answers to direct questions. Engineering-fluent peer level, not teaching level. The user types in lowercase, often with typos; do not pattern-match on perfect grammar.
- **Recovery points are a standard, not optional.** Before risky operations, take a snapshot: git tag (`pre-batch-X-YYYYMMDD-HHMMSS` and `post-batch-X-...`), KB DB copy, or both. The user explicitly asked for this on Phase 16 Batch D ("Ship D but give me the option to revert to this exact point"). Always offer a revert path with a concrete command before shipping.
- **Auto mode means execute, but ASK on ambiguity.** When auto mode is active, default to action on low-risk work. If multiple reasonable paths exist or the user's message is ambiguous, surface options in 1-2 sentences and let them pick — don't burn an hour the wrong direction. If the user says "1" or a single token, they're answering the option list you just offered.
- **Stress-testing IS the engineering work.** After each batch ships, run 2-3 diverse real scenarios end-to-end (not just unit tests). Phase 16 Batch A's stress test surfaced two bugs (Phase 15e snapshot scope, single-call inference time) that unit tests would never have caught.

Do NOT use `until <check>; do sleep N; done` patterns inside Bash calls — they leave orphan bash.exe/sleep.exe/conhost.exe processes on Windows that visibly spawn terminal windows. The user noticed and was annoyed in the 2026-05-06 session ("there's a 'terminal' baby being spawned every 5 seconds"). For periodic checks, check ONCE per user prompt instead. If the loop ends or you stop polling, SAY SO explicitly — claiming to monitor when you aren't will get you called out.
- **The vision.** "The ultimate AI assistant offline LLM." Search indexes (FTS5 + nomic embeddings + reranking) overcompensate for the small model + 4GB GPU constraint. Every phase moves toward Qwen-only operation; Claude is the ceiling that should fade to zero as Qwen earns trust through graduation tests + auto-pinning + (eventually) Claude-skip on high-trust patterns.

Always check `/kb show <id>` per-component breakdown (file Jaccard / tool Jaccard / step proximity) before declaring a problem. ## Phase 18 — /gwen open-system + /gwenask in-house authoring (2026-05-07)

- **`/gwen` is a literal-recipe dispatcher, NOT an agent.** Reads `update.message.text` directly (preserves newlines), executes STEP-N tool calls verbatim. No brain, no KB, no orchestrator, no LLM in the dispatch path. Fast-path detection: if first line matches `STEP \d+:`, parse + run. Tools: `read_file`, `list_dir`, `write_file`, `edit_file`, `run_bash`, `done`. Paths via `Path.expanduser()` — no sandbox.
- **`/gwenask` lets local Qwen author recipes for stdlib tkinter apps.** User describes the app in English; Qwen produces a 5-step recipe (mkdir → write_file → PyInstaller → copy .exe → done). Server-side wrapping converts plain `content="<source>"` to `content_b64gz="<gzipped>"` so the recipe survives Telegram paste-mangle (quote-strip, smart-quote conversion, soft-wrap). Qwen can't compute base64 itself; the wrap happens AFTER generation.
- **Telegram paste-mangle is the load-bearing constraint.** Telegram clients strip outer double-quotes inconsistently across mobile/desktop, occasionally smart-quote-convert, and soft-wrap long lines. b64gz alphabet (`[A-Za-z0-9+/=]`) survives all three. EVERYTHING in the recipe pipeline assumes content args may arrive with quotes stripped or newlines inserted; the parser has fallbacks for each.
- **Step-aware quote-tolerant wrapper.** `SentinelTelegramBot._wrap_plain_content_as_b64gz` splits the recipe on STEP boundaries FIRST, then anchors at `content="` and consumes everything to chunk-end (no first-`"` truncation). Qwen's frequent `text=f"..."` (unescaped inner double-quote) used to silently truncate the source mid-line; the new wrapper packages everything to the next STEP marker regardless of inner quoting.
- **Executor MUST hard-fail on any step error (Phase 18-fix 2026-05-07).** `core/gwen_agent._execute_recipe` previously logged WARNING and continued past tool errors. That allowed write_file to silently fail then PyInstaller + copy to run against stale source from a prior build, returning false ✅. Both `aborted_unparseable` AND `result["error"]` paths now BREAK the loop. Final summary surfaces the failure clearly. Without this, "✅ Done" cannot be trusted.
- **AST + name-consistency validation gate.** `_validate_recipe_source` decodes b64gz, runs `ast.parse`, walks `ClassDef` nodes vs module-level `Name(...)` calls. Catches: SyntaxError, undefined-name instantiation (e.g. `App(root)` with `class TicTacToe:`), unclosed parens. `handle_gwenask` does ONE retry with the specific error fed back as `PREVIOUS ATTEMPT FAILED VALIDATION: <error>` — more retries don't help. This single mechanism caught 4+ distinct Qwen authoring bugs without needing per-bug prompt rules.
- **Worked examples beat abstract rules.** Qwen pattern-matches the example shape over the rule text. Every prompt rule that needed structural compliance got paired with a corrected worked example (TIMER, EVENT-DRIVEN, GRID — all using `self.root` consistently). When the Counter example used `tk.Button(root, ...)` instead of `tk.Button(self.root, ...)`, Qwen replicated the bug; rewriting the example fixed it.
- **`num_predict=6144` on /gwenask Qwen calls.** Ollama default ~1024 tokens was truncating Qwen mid-class for any recipe over ~2KB. `OllamaClient.generate` now accepts `num_predict: int | None`; `_qwen_generate` forwards it; `handle_gwenask` passes 6144.
- **`/gwenask` GWENASK_SYSTEM is ~5KB.** Sections in order: STRICT OUTPUT RULES → RECIPE-QUOTING DISCIPLINE → TKINTER IMPORTS → TKINTER ESSENTIALS (8 numbered rules) → FILE I/O ESSENTIALS → PICK THE RIGHT PATTERN (TIMER vs EVENT-DRIVEN) → 3 worked mini-examples → 5-step recipe shape boilerplate. Qwen 8K context budget leaves ~3KB for output (recipes typically 1.5-3KB). Headroom is thin; if context grows further, consider externalizing rules to PROMPT_BRIEF.md and only loading the relevant pattern.

## What NOT to Do

- Do not use `exec()` or `eval()` in the main process. Code execution goes through subprocess.
- Do not hardcode paths. Import from config.
- Do not add LLM calls to the router or orchestrator.
- Do not create skills that bypass the registry. Every skill must inherit BaseSkill and live in `skills/`.
- Do not install unpinned dependencies.
- Do not use in-memory flags for state that needs to survive crashes. Use the DB.
- Do not let the scheduler enforce its own GPU serialization (Phase 11). Always route GPU-bound jobs through `route()` and rely on the worker's DB lock; the scheduler's `SCHEDULER_MAX_CONCURRENT` is a sanity belt only.
- Do not bind the health endpoint to anything other than `127.0.0.1` (Phase 11). Exposing it externally leaks queue + GPU + memory state without auth.
- Do not call into `/internal_*` from outside the scheduler (Phase 11). They're not auth-gated and assume internal callers; user-facing maintenance lives in real `/commands`.
- **NO AUTO-COMMITS** (owner directive 2026-05-06). The ONLY path that performs `git commit` is the manual `/commit` Telegram command. No skill, agent, scheduler job, graduation step, or test may run `git commit` on the user's working tree. `_git_commit_for_graduation` (Phase 15e) is disabled as a no-op; `_git_commit_changes` (Phase 10) was already a no-op. Tests that previously made real commits (`tests/test_code_assist.py`, `tests/test_pipeline_phase9.py`) are NOT safe to run unattended — they bypassed this rule pre-directive and require redesign before re-running. Trade-off: graduation's tree-state discipline (Phase 15e) loses its commit boundary and falls back to stash-only restoration, which can lose work if `git stash push -u` silently fails. Phase 17 candidate: redesign graduation to never need a commit (e.g., serialize dirty diff to temp file before reset, re-apply after).
